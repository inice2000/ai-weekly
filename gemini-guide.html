第二章

安裝準備

只需要安裝兩個東西：Node.js 和 Gemini CLI。

寫給零基礎讀者

什麼是終端（Terminal）？

在這份手冊裡，你會經常看到「在終端中輸入命令」。終端（Terminal）就是一個純文字的操作介面，你輸入一行指令，電腦就執行對應的操作——可以理解為「用打字代替滑鼠點選」。

macOS 自帶終端應用，叫「Terminal」。你也可以安裝更好用的 iTerm2（免費），但系統自帶的完全夠用。

如何開啟終端
方法一（推薦） 按 Cmd + 空格 開啟 Spotlight，輸入 Terminal，按回車
方法二 開啟 Finder → 應用程式 → 實用工具 → 終端

開啟後你會看到一個白色或黑色視窗，游標在閃爍等待輸入。直接把手冊中的命令複製貼上進去，按回車執行就行了。

步驟 1

安裝 Node.js

Gemini CLI 需要 Node.js 22 或更高版本。macOS 推薦用 Homebrew 安裝：

Terminal
# 如果還沒裝 Homebrew，先裝它（一行命令）
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# 用 Homebrew 安裝 Node.js
brew install node

# 驗證安裝
node --version   # 應顯示 v22.x.x 或更高
💡
沒裝過 Homebrew？

Homebrew 是 macOS 的軟體包管理器，相當於 Mac 上的「應用商店命令列版」。上面的安裝命令在終端裡貼上執行就行，會自動完成所有設定。

也可以直接下載安裝包：訪問 nodejs.org，下載 macOS 版安裝包，雙擊安裝即可。

步驟 2

安裝 Gemini CLI

開啟「終端」應用（按 Cmd + 空格，輸入 Terminal，回車），執行：

Terminal
npm install -g @google/gemini-cli

# 驗證安裝
gemini --version

看到版本號說明安裝成功。

第四章

建立專案 · 搭建檔案結構

給你的助手安個「家」，所有記憶和規則都存在這裡。

步驟 1

安裝 Git

Git 是版本管理工具，用來保護你的助手記憶和配置不會丟失。macOS 不一定自帶，先檢查一下：

Terminal
git --version

如果看到版本號（如 git version 2.x.x），說明已經裝好了，跳到步驟 2。

如果提示「command not found」或彈出安裝對話方塊，有兩種安裝方式：

方式一：Xcode Command Line Tools（推薦，彈出對話方塊時直接點安裝）
xcode-select --install
方式二：Homebrew（如果已裝 Homebrew）
brew install git
💡
Git 有什麼用？

Git 會記錄你助手所有配置檔案的修改歷史。萬一改錯了，可以隨時回退到之前的版本。就像文件的「撤銷」功能，但可以回到幾天甚至幾個月前的狀態。

步驟 2

建立專案資料夾

在你喜歡的位置建立資料夾，比如放在使用者目錄下：

Terminal
# 建立專案資料夾
mkdir ~/myAgent
cd ~/myAgent

# 初始化 Git 倉庫
git init
⚠️
如果是第一次用 Git

需要設定你的名字和郵箱（只需執行一次，以後不用再設定）：

Terminal · 首次 Git 配置
git config --global user.name "你的名字"
git config --global user.email "你的郵箱@example.com"
步驟 3

搭建檔案結構

建立所有必要的資料夾：

Terminal
mkdir -p config docs/lessons docs/reports docs/others memory/tasks

完成後，你的專案結構如下：

myAgent/
├── config/ # 身份配置
├── docs/
│   ├── lessons/ # 踩坑記錄·經驗教訓
│   ├── reports/ # 故障處理記錄
│   └── others/ # 調查結果·設計備忘
├── memory/
│   └── tasks/ # 各任務的上下文狀態
└── GEMINI.md # 核心指令書（下一步建立）
步驟 4

連線 GitHub（雲端備份，推薦）

將助手的配置和記憶備份到 GitHub，換電腦一分鐘恢復。

  1. 訪問 github.com，註冊或登入賬號
  2. 點選右上角「+」→「New repository」
  3. 倉庫名填 myAgent，選擇 Private（私有）
  4. 不要勾選「Add a README file」，直接點「Create repository」
🔒
必須選 Private（私有）

倉庫裡包含你的助手配置和記憶檔案，選 Public 會被所有人看到。

完成下一章「一鍵初始化」後，執行以下命令推送到 GitHub：

Terminal · 首次推送
cd ~/myAgent
git remote add origin https://github.com/你的使用者名稱/myAgent.git
git add .
git commit -m "初始化 AI 助手配置"
git push -u origin main
# 首次推送會彈出 GitHub 登入視窗，授權即可
日常同步

助手積累新經驗後同步一次

Terminal
cd ~/myAgent
git add .
git commit -m "更新助手配置和經驗記錄"
git push
換電腦時

一鍵恢復所有配置

新電腦裝好 Node.js 和 Gemini CLI 後，不需要重新配置，直接從 GitHub 恢復：

Terminal
git clone https://github.com/你的使用者名稱/myAgent.git ~/myAgent
cd ~/myAgent
gemini
# 所有配置、記憶、歷史經驗全部自動恢復

第五章

一鍵初始化

把下面的內容複製貼上到 Gemini 對話方塊，它會自動建好所有檔案。

確保你在 ~/myAgent 目錄下，啟動 Gemini 後，把下面整段內容完整複製貼上到對話方塊：

Bootstrap Prompt · 複製貼上到 Gemini 點選複製全部 
請幫我完整搭建一套 AI 助手系統。按以下要求建立所有檔案和目錄，不需要問我確認，直接全部執行完成。

## 第一步：建立 GEMINI.md（專案根目錄）

建立檔案 `GEMINI.md`，內容如下：

```markdown
# GEMINI.md

**預設回覆語言：繁體中文**

## 身份

我叫**小星**，是主人的自動化助手。

啟動後請閱讀 `config/identity.md`，準備好響應身份確認暗號：「小星，你在嗎？」

## 必須遵守的核心規則

1. 始終遵守 `docs/rules.md` 中定義的規則
2. 開始任務前，查閱 `docs/lessons/` 中的歷史經驗
3. 完成任務後，將經驗按分類記錄到 docs/ 目錄
4. rm、git reset --hard 等危險操作必須先向使用者確認
5. 不得將密碼、API Key 等寫入普通文件

## 工作流程

1. 接收任務 → 2. 查閱歷史經驗 → 3. 執行 → 4. 記錄經驗

## 行為準則

- 優先修改已有檔案，而非新建檔案
- 保持輸出簡潔、結構清晰
- 不確定的事情要問，不要猜測
```

## 第二步：建立 config/identity.md

```markdown
# 身份配置

## 名字
**小星**

## 暗號（使用者說）
小星，你在嗎？

## 回應（必須這樣回答）
在。我是小星，主人的自動化助手。
我記得：預設中文回覆、危險操作先確認、任務前讀經驗文件、任務後寫總結。
需要我做什麼？

## 人格核心
| 屬性 | 內容 |
|------|------|
| 名字 | 小星 |
| 語言 | 預設繁體中文 |
| 角色 | 貼身自動化助手 |
| 底線 | 破壞性操作必須確認 |
```

## 第三步：建立 docs/rules.md

```markdown
# 核心規則

## 安全規則（最高優先順序）
- 禁止自動執行：rm -rf、git reset --hard、git push --force
- 不得將密碼、API Key 列印在輸出中
- 只在使用者授權的目錄內操作

## 工作規則
1. 優先修改已有檔案
2. 不確定的事情要問
3. 遇到錯誤先查 docs/lessons/

## 文件記錄規則
| 情況 | 更新目標 |
|------|----------|
| Bug/踩坑 | docs/lessons/YYYY-MM-DD/ |
| 故障 | docs/reports/YYYY-MM-DD/ |
| 調查/設計 | docs/others/YYYY-MM-DD/ |
```

## 第四步：建立 memory/MEMORY.md

```markdown
# 助手記憶索引
- 使用者檔案：見 `memory/user_profile.md`
- 任務狀態：見 `memory/tasks/{任務名}/context.md`
```

## 第五步：建立 memory/user_profile.md

```markdown
# 使用者檔案
（使用過程中由助手逐步填寫）
- 職業：
- 常用工具：
- 語言偏好：繁體中文
```

## 第六步：驗證

所有檔案建立完成後，列出目錄結構確認檔案都存在，然後回覆：
「所有檔案已建立完畢。請輸入暗號驗證：小星，你在嗎？」
✏️
記得改名字

把「小星」和「主人」改成你想要的名字！這是你助手的個性化身份。

檔案整理與分析

🖥️

自動開啟網頁、填寫表單、截圖儲存、抓取頁面內容……重複性網頁操作不用再手動。

🔍

調查與資訊整合

讓助手搜尋某個主題的資料，整理成結構化摘要，附帶來源——比自己逐個搜尋快得多。

💡

與其說「幫我寫個指令碼」，不如說「幫我寫一個指令碼，把桌面上所有 .jpg 檔案按年月移動到對應子資料夾裡」。描述得越具體，助手產出越精準。

對話示例

你說什麼助手會做什麼
幫我整理桌面檔案，按型別分資料夾寫指令碼自動分類整理

這次遇到了個問題，幫我記錄一下自動寫入 docs/lessons/ 當日檔案

跨對話儲存任務進度（重要）

⚠️
Gemini CLI 每次對話是獨立的，不會自動記住上次內容。

解決方法：結束對話前說「更新任務上下文」，助手會儲存進度到 memory/tasks/{任務名}/context.md。下次說「讀取 XX 任務上下文，繼續上次的工作」即可恢復。

可用命令（Skill）

你的助手預置了幾個常用命令，在對話中直接輸入即可觸發：

命令用途什麼時候用
/start-task任務開始檢查清單開始一個新任務時
/finish-task任務完成・回顧總結任務做完後
/review程式碼・作業內容審查想讓助手檢查你的工作時
/research主題調查・資訊收集需要調查某個話題時
/lesson即時記錄經驗教訓踩坑後想立刻記錄時
⚠️
重要：Gemini CLI 沒有原生 Skill 系統

與 Claude Code 不同，Gemini CLI 不會自動載入 skill 檔案。它是透過讀取 GEMINI.md 中的對映表，然後自己去讀取對應檔案來模擬執行的。偶爾可能會忘記讀檔案或省略步驟。

如果助手沒有按預期執行，可以明確提醒：「請先讀取 skills/start-task.md，然後嚴格按步驟執行」。

讓助手記住事情

直接跟助手說，它會更新 memory/ 下的檔案：

  • 「記住我的工作時間是早上 10 點到晚上 8 點」
  • 「記住專案程式碼風格用 4 空格縮排」

讓助手總結經驗

任務完成後說：

  • 「把這次的經驗記錄一下」
  • 「這次踩了個坑，幫我記錄到教訓文件」

→
「繼續做 XX 任務」
→

→
遇到問題 → 查 lessons
→

第七章

五大核心工作原則

讓助手越來越聰明的秘訣。

1

指令碼優先

能用指令碼（程式碼）實現的操作，絕不依賴純對話。AI 只負責做判斷，結果用結構化資料輸出，實際檔案操作由指令碼執行。

實際含義：批次處理檔案時，讓助手寫一個指令碼來跑，而不是一個一個操作。更穩定，也可以複用。
2

經驗積累

每次遇到新問題解決後，必須記錄到 docs/lessons/。下次遇到同類問題，先查文件，不重複踩坑。

實際含義：把助手當成會學習的同事，每次解決問題後讓它「寫總結」，慢慢積累針對你工作場景的經驗庫。
3

任務上下文儲存

每個長期任務在 memory/tasks/{任務名}/context.md 儲存當前狀態。新對話開始時讀取這個檔案，恢復任務進度。

實際含義：每次對話是獨立的，不會自動記住上次內容。透過 context.md 檔案讓助手在新對話中快速「想起來」上次做到哪了。
4

危險操作二次確認

所有不可逆操作（刪除、覆蓋、重置）執行前必須明確告知使用者並等待確認。即使使用者之前說過「你可以直接做」，也要再次確認。

實際含義：防止助手誤操作導致資料丟失。寧可多問一句，不要出事後悔。
5

邊界清晰

不確定的事情要問，不要猜測後直接執行。比如：「我要刪除資料夾裡的圖片」——是全部？還是某一類？先確認。

實際含義：AI 犯錯的代價可能很大，多問一句只需要幾秒鐘。

第八章

常見問題 · 三大 Agent 對比

遇到問題？先來這裡找找答案。

Q終端提示「gemini: command not found」
  1. 關閉終端視窗，重新開啟一個
  2. 如果還是不行，重新執行：npm install -g @google/gemini-cli
  3. 檢查 Node.js 版本：node --version（需要 v22+）
Q登入時瀏覽器沒有彈出來

檢視終端輸出中是否有登入連結，手動複製到瀏覽器開啟。確認你的 Google 賬號可以正常登入。

Q提示 HTTP 錯誤或連線超時

Gemini CLI 有時會出現隨機 HTTP 錯誤，這是目前已知的穩定性問題。

  1. 等幾秒鐘後重試
  2. 如果持續報錯，退出後重新啟動 gemini
  3. 確認網路正常（能正常訪問 Google 網站）
Q助手省略程式碼或把任務推回給我

這是 Gemini CLI 的已知行為特點。應對方法：

  1. 明確要求：「請寫出完整程式碼，不要省略」
  2. 把大任務拆分成小步驟，一步一步來
  3. 如果問題持續，這個特定任務可以換用 Claude Code 或 Codex CLI
Q助手忘記了上次的對話內容

這是正常現象。Gemini CLI 每次對話是獨立的。

  1. 開始時告訴助手：「請讀取 memory/tasks/XX任務/context.md，繼續上次的工作」
  2. 養成習慣：每次結束對話前，讓助手更新 context.md
Q真的完全免費嗎？

是的，個人 Google 賬號完全免費。有每分鐘請求數限制（rate limit），高頻使用時可能遇到限速，但日常使用完全夠用。

Q想把助手名字改成別的

用任意文字編輯器（VS Code、TextEdit 等）開啟 GEMINI.md 和 config/identity.md，把所有「小星」替換成你想要的名字，儲存後重啟 Gemini CLI 即可。

Q如何讓助手操作瀏覽器？

安裝 Playwright 後即可實現瀏覽器自動化：

Terminal
npm install -g playwright
npx playwright install

之後告訴助手：「幫我開啟瀏覽器，訪問 XX 網站，截圖儲存」。

QGEMINI.md 和其他配置檔案能修改嗎？

完全可以。GEMINI.md 是助手的「行為準則」，你隨時可以增加新規則、修改工作偏好。改完後重啟 Gemini CLI 生效。推薦用 VS Code 編輯（免費下載：code.visualstudio.com）。
